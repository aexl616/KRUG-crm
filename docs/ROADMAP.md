# KRUG CRM Roadmap

## Vision

**KRUG CRM — операционная система музыкальной студии КРУГ.**

Это не просто таблица клиентов и не бухгалтерия. Это внутреннее приложение, из которого сотрудники студии управляют:

- расписанием
- записями
- клиентами
- услугами
- сотрудниками
- оплатами
- выплатами
- бюджетом
- отчётами
- будущей онлайн-записью
- будущим Telegram-ботом

Главный объект системы — **Запись**.

```text
Запись → Клиент → Сотрудник → Услуга → Платёж → Отчёт
```

---

## Development rules

### One patch = one main idea

Один патч должен решать одну крупную задачу. Не смешивать календарь, роли, Telegram, аналитику и редизайн в одном патче.

### 3 questions before every patch

Перед каждым патчем отвечаем:

1. Какую проблему студии решает патч?
2. Что станет быстрее или проще после патча?
3. Почему этот патч нужен сейчас, а не позже?

### 4-year rule

Перед добавлением функции спрашиваем:

> Будет ли эта функция полезна студии в ближайшие 4 года?

Если нет — откладываем.

### Booking-first logic

Запись — центр всей системы. Все крупные функции должны усиливать рабочую цепочку студии, а не просто добавлять ещё один экран.

## 🟡 Patch 0.4 — Studio Calendar Core

**Тип:** ⚫ Core + 🟢 Feature  
**Статус:** In progress / stabilization

### Problem

Администратор должен видеть расписание студии и управлять записями из одного места.

- смена статуса

- выбор услуги

### Why now

Без стабильной логики записей нельзя нормально делать Telegram-бота, онлайн-запись, роли, аналитику и облако.

### Scope

- недельный календарь
- вкладка “Записи”
- статусы записей
- создание записи из кнопки
- создание записи кликом по слоту
- редактирование записи
- правый сайдбар деталей записи
- связь записи с клиентом
- связь записи с сотрудником
- связь записи с услугой
- автосоздание платежа при завершении
- защита от дублей платежей
- CRM только для сотрудников

### Current follow-up

**Тип:** 🔴 Fix + 🔵 UX

## ⬜ Patch 0.8 — Roles & Access

**Тип:** ⚫ Core

### Problem

Разные сотрудники не должны видеть одни и те же данные.

### What becomes faster / safer

Можно давать доступ сотрудникам без риска, что они увидят выручку, прибыль, расходы или настройки.

### Why after calendar/resources

Права доступа лучше накладывать на уже понятную структуру разделов и сущностей.

### Planned roles

- Владелец
- Администратор
- Звукорежиссёр
- Исполнитель
- Дизайнер
- Продюсер
- Другое / гость

### Access model draft

#### Владелец

Полный доступ ко всем разделам и данным.

#### Администратор

- Сегодня
- Календарь
- Записи
- Клиенты
- Платежи
- Каталог услуг
- Сотрудники
- операционные настройки

Ограничить доступ к глубокой прибыли/бюджету, если нужно.

#### Звукорежиссёр

Видит:

- свой календарь
- свои записи
- клиентов своих сессий
- свои выплаты, если нужно

Не видит:

- общую выручку студии
- прибыль
- расходы
- бюджет
- выплаты других сотрудников
- редактирование услуг
- редактирование сотрудников

#### Исполнитель / дизайнер / продюсер

Видит только назначенные работы и связанные записи/клиентов.

### Important note

Пока CRM на localStorage, роли будут визуальным ограничением, а не настоящей безопасностью. Настоящая безопасность появится с backend/cloud.
